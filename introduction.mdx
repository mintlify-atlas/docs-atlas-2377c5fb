Kubespray is a collection of Ansible playbooks that deploys and manages production-ready Kubernetes clusters on any cloud provider or bare metal infrastructure. It combines Ansible's generic OS configuration management with kubeadm's Kubernetes lifecycle management to give you a composable, multi-platform deployment tool that works on AWS, GCE, Azure, OpenStack, vSphere, Equinix Metal, and bare metal alike.

## What Kubespray does

Kubespray handles the full cluster lifecycle: initial provisioning, node scaling, upgrades, and cluster reset. You define your infrastructure in an Ansible inventory file, adjust variables in `group_vars`, and run a playbook. Kubespray takes care of installing container runtimes, configuring networking plugins, bootstrapping etcd, and standing up the Kubernetes control plane.

## Comparison with other tools

### Kubespray vs kubeadm

[kubeadm](https://github.com/kubernetes/kubeadm) provides domain knowledge of the Kubernetes cluster lifecycle — self-hosted layouts, dynamic discovery services, and cluster bootstrapping primitives. Kubespray handles generic OS configuration tasks from the Ansible world, plus initial Kubernetes clustering (with networking plugins included) and control plane bootstrapping.

Kubespray has used kubeadm internally for cluster creation since v2.3, consuming its lifecycle management expertise while offloading OS-level configuration. This means you get the benefits of both tools.

### Kubespray vs kops

[kops](https://github.com/kubernetes/kops) performs provisioning and orchestration itself and is more tightly integrated with the unique features of the clouds it supports. Kubespray runs on bare metal and most clouds, using Ansible as its substrate — making it less opinionated about where you deploy.

If you work with Ansible already, run clusters on multiple platforms, or need bare metal support, Kubespray is the better fit. If you only target a single cloud and want tighter native integration, kops may be preferable.

### Network plugins

Kubespray supports ten CNI plugins. The default is **Calico**, except when using Vagrant where **Flannel** is the default.

| Plugin | Description |
|---|---|
| [Calico](https://docs.tigera.io/calico/latest/about/) v3.31.5 | Flexible networking and network policy, supports BGP and overlay modes |
| [Cilium](https://cilium.io/) v1.19.3 | eBPF-based L3/4/7 networking with kernel-level visibility |
| [Flannel](https://github.com/flannel-io/flannel) v0.28.4 | Simple gre/vxlan layer 2 overlay networking |
| [Kube-OVN](https://github.com/alauda/kube-ovn) v1.12.21 | OVN-based network virtualization for enterprises |
| [Kube-router](https://github.com/cloudnativelabs/kube-router) v2.1.1 | L3 CNI using IPVS, iptables, and BGP |
| [Multus](https://github.com/k8snetworkplumbingwg/multus-cni) v4.2.2 | Meta-plugin enabling multiple network interfaces per pod |
| [kube-vip](https://github.com/kube-vip/kube-vip) v1.0.3 | Virtual IP and load balancing for control plane |
| CNI plugins | v1.9.1 |
